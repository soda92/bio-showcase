# Frontend Debugging Guide: Deno Fresh & Preact

This guide explains how to debug the Deno Fresh frontend of the Bioinformatics Workbench. Because Deno Fresh is a server-side pre-rendered framework, code executes in two distinct runtimes: the **Deno Backend Server** (SSR) and the **Client Browser** (Interactive Islands).

---

## 1. Server-Side Debugging (VS Code)

Any code that runs on the server during pre-rendering or backend request routing is executed by the Deno runtime.

### What runs on the server?
*   Route files (`frontend/routes/*.tsx`)
*   Global layouts and apps (`_app.tsx`)
*   Data fetching calls before rendering

### How to debug:
1.  **Set breakpoints**: Open any route (e.g., [about.tsx](file:///home/soda/src/bio-showcase/frontend/routes/about.tsx)) or server-side component in VS Code and click to the left of the line numbers to set a breakpoint.
2.  **Start debugging**:
    *   Open the Run & Debug panel in VS Code (`Ctrl+Shift+D`).
    *   Select **Deno Frontend** from the dropdown configuration.
    *   Press `F5` to launch.
3.  **Inspect**: Open `http://localhost:8000` in your browser. When the page is requested, VS Code will halt execution at your breakpoint, allowing you to inspect local variables, call stacks, and step over/into functions.

> [!NOTE]
> We have configured the VS Code debugger configuration to launch Deno with the `--inspect` and `--watch` flags. The server runs immediately on launch and automatically hot-reloads on file changes without disconnecting your debugging session.

---

## 2. Client-Side Debugging (Browser DevTools)

Interactive components (Islands) are shipped as JavaScript to the browser and execute in the user's browser runtime.

### What runs on the client?
*   Island files (`frontend/islands/*.tsx`)
*   Preact lifecycle hooks (`useState`, `useEffect`) inside islands
*   Browser events (button clicks, form submits, editor inputs)

### How to debug:
1.  **Open Developer Tools**: Open the workbench page in your browser and press `F12` (or `Ctrl+Shift+I` / `Cmd+Option+I` on Mac).
2.  **Find the code**:
    *   Select the **Sources** tab (Chrome/Edge) or **Debugger** tab (Firefox).
    *   Press `Ctrl+P` (or `Cmd+P` on Mac) to search for files.
    *   Type the name of your island (e.g., `SequenceAlignment.tsx` or `PrimerDesign.tsx`) and open it.
3.  **Set breakpoints**: Click on the line numbers inside the browser editor (e.g., inside an event handler like `handleSolve` or `handleSubmit`).
4.  **Inspect**: Perform the action in the web UI (like clicking "Solve Alignment"). The browser will pause execution, letting you hover over variables, evaluate statements in the Console tab, and inspect the call stack.

---

## 3. Recommended Workflow

| Task | Runtime | Debugging Tool |
| :--- | :--- | :--- |
| Inspecting server database queries or initial renders | Deno Server (SSR) | VS Code debugger (`F5`) |
| Debugging CodeMirror mounting and text edits | Browser Client | Browser DevTools (`F12` Sources) |
| Debugging alignment dynamic programming grid hover effects | Browser Client | Browser DevTools (`F12` Sources) |
| Analyzing API errors or request payloads | Both | Browser DevTools (Network tab) & Server Terminal |

## 4. Crucial Tips & Gotchas

### ⚠️ The "SSR Debugger Trap" (Backend vs. Frontend Execution)
Because Deno Fresh pre-renders HTML on the server and makes it interactive in the browser, components inside the `islands/` directory execute in **both** runtimes.
*   **The Problem**: If you set a breakpoint inside a button click handler (like `FillDemoData`) using the **Deno Backend Debugger** (the green Node.js/Deno DevTools window, or VS Code's `F5`), the debugger **will not pause**. This is because button clicks and user interactions execute entirely inside the client's browser engine.
*   **The Rule**: 
    *   Set breakpoints in **VS Code / Remote Deno DevTools** to debug loader scripts, server-side pre-renders, and APIs.
    *   Set breakpoints in your **Main Browser DevTools (F12)** to debug user clicks, reactive states (`useState`, `useEffect`), and interactive island logic.

### 📝 Italic vs. Normal Filenames in DevTools
In Chrome DevTools, you will notice files in the tab bar and source tree displayed in different styles:
*   **Italic File Names (`*about.tsx*`) = Virtual Source-Mapped File**: This is your original TypeScript/JSX source code. V8 doesn't execute this directly, but DevTools maps execution to it. Use this for reading code and setting your breakpoints.
*   **Normal File Names (`about.tsx` or compiled JS) = Real Executed Script**: This is the actual compiled JavaScript code generated by Deno and run inside the V8 heap.
*   **DevTools Tip**: To keep a file open and prevent Chrome from replacing it when you navigate other files, **double-click the tab header** or **double-click the file in the file tree** to pin it (changing it from italicized preview to normal text).
